# Migration: 0.7 to 1.0

This guide covers the schema and API expectations for moving from the 0.7 line
to the 1.0 contract.

## Database Schema

Existing durable stores need migration support for the reliability fields added
to the outbox and inbox models.

Outbox records need durable dispatch state:

- `status`: `pending`, `processing`, `published`, `failed`, or `dead_letter`
- `nextRetryAt`
- `lastError`
- `lockedBy`
- `lockedUntil`
- `publishedAt`

Inbox records need idempotency and dead-letter state:

- `messageId`
- `dedupeKey`
- `status`: `pending`, `processing`, `processed`, `failed`, or `dead_letter`
- `lastError`
- `processedAt`

Replace old inbox uniqueness on `(topic, group, messageId)` with uniqueness on
`(group, dedupeKey)`. Keep `messageId` for broker traceability, but use
`dedupeKey` for the stable idempotency identity. The first-party MikroORM
entity stores `dedupeKey` as a 512-character string so the composite unique
index stays portable across PostgreSQL and MySQL.

## Publish and Transaction Behavior

The stable publish API is:

```ts
publish(topic, payload, { headers, tx, immediate }?)
```

When `tx` is provided without `immediate: true`, CAP writes only the outbox row
inside the transaction. The scheduler publishes after commit. `immediate: true`
emits immediately and is intentionally non-atomic with the database commit.

## Storage and Driver Requirements

Production multi-instance outbox dispatch requires atomic claim/lease behavior.
The MikroORM adapter uses pessimistic partial write locking on lock-capable SQL
drivers such as PostgreSQL and MySQL.

SQLite and other local/non-locking drivers are suitable for demos, development,
and single-process tests only. They are not supported for multi-instance
durable outbox dispatch.

The first-party multi-instance DB gate covers PostgreSQL and MySQL. SQL Server
is not multi-instance supported by the first-party MikroORM adapter until it has
a SQL Server-specific claim implementation.

## Module Registration

`CapModule` remains global for v1 and should be registered once at the
application root:

```ts
CapModule.forRoot({
  imports: [MikroStorageModule, transportModule],
});
```

Adapter modules must export the CAP storage and transport tokens they provide.
Legacy adapter factory registration helpers are not part of the v1 contract.
